# Documentation Best Practices

## Definition

A set of proven principles for creating documentation that is accurate, useful, discoverable, and maintainable. These practices apply across product docs, developer docs, process docs, and personal knowledge bases.

## The 10 Core Best Practices

**1. Lead with the answer**
- Put the core explanation near the top of the page
- Don't bury the most important details
- Help readers get what they need without searching through the entire page

**2. Write for a specific audience**
- A setup guide for customers shouldn't read like architecture documentation for engineers
- Match the tone, detail level, and terminology to your intended audience
- Spell out unfamiliar terms; avoid unnecessary jargon

**3. Keep one page focused on one topic**
- Avoid pages that try to explain everything at once
- Use links to point people to related content and next steps
- Makes pages easier to scan and maintain

**4. Link related pages**
- Signpost next steps at every stage
- Help users continue their journey without starting over
- Create a web of interconnected knowledge

**5. Use descriptive headings**
- Headings should match the questions people actually search for
- Improve scanning, AI-readability, and SEO
- Make structure clear at a glance

**6. Prioritize examples over abstraction**
- Show what good looks like with concrete examples
- Include screenshots, code samples, diagrams, or sample scenarios
- Help readers understand when they've successfully completed a task

**7. Make troubleshooting easy to find**
- People often visit docs when something breaks
- Make FAQs clear, accessible, and easy to search
- Include common errors, edge cases, and recovery steps

**8. Use consistent terminology**
- Don't switch between different names for the same feature or concept
- Define terms once, then use them consistently throughout
- Reduces confusion and improves scannability

**9. Make your docs accessible**
- Add alt text to images
- Use readable structure and scannable formatting
- Ensure content is usable by all readers

**10. Review and update regularly**
- Stale documentation loses trust fast
- Build docs updates into your release/maintenance process
- Carry out regular reviews to catch outdated information

## How These Apply to Diátaxis

**Diátaxis Framework:** Tutorials (hands-on), How-to Guides (solve problems), Explanations (understand concepts), Reference (lookup facts)

- **Lead with the answer** → Tutorials show immediate, successful action; References prioritize lookup speed
- **Write for a specific audience** → Each Diátaxis type serves different reader goals; tutorials for beginners, references for experienced users
- **One topic per page** → Each Diátaxis page has one clear purpose and reader goal
- **Link related pages** → Diátaxis pages naturally cross-reference: tutorials link to explanations for deeper learning
- **Descriptive headings** → Critical in Diátaxis for navigation between types
- **Examples over abstraction** → Tutorials and how-to guides are driven by examples; explanations provide the "why"
- **Troubleshooting** → How-to guides often include troubleshooting as a natural part of solving problems
- **Consistent terminology** → Essential across all four Diátaxis types to avoid confusion
- **Accessible** → Applies universally; clear Diátaxis structure improves accessibility
- **Regular updates** → Applies to all documentation types

## Key Synthesis

The best practices are **universal** — they apply to professional documentation, personal wikis, Diátaxis-structured docs, and any form of knowledge transfer. The core principle is the same across all contexts:

> **Remove friction between the reader's question and the solution.**

Different documentation types (Diátaxis) and different audiences require different emphasis on these practices, but none of them contradict. A well-written tutorial leads with examples (example > abstraction), uses consistent terminology, and links to explanations for readers who want deeper understanding.
