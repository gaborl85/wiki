# Technical Documentation

## What is it?

Technical documentation is written content that explains a product, system, API, process, or codebase. It helps people build, use, maintain, and troubleshoot software by answering three core questions: "What is this?", "How does it work?", and "What do I do next?"

## Key Characteristics

- **Clear and specific** — Explains complex topics without unnecessary jargon
- **Easy to scan** — Uses headings, examples, and visual formatting to guide readers
- **Audience-aware** — Tailored to the specific needs and background of its readers
- **Task-focused** — Helps people complete real work, not just understand concepts
- **Maintained** — Kept current as products and processes evolve
- **Searchable** — Structured to be discovered through search and navigation

## Main Audiences

**Developers**
- Need: API docs, architecture notes, SDK guides, source code documentation
- Goal: Build on top of systems, integrate with other tools, understand technical decisions

**Customers**
- Need: Setup guides, tutorials, troubleshooting steps, FAQs, release notes
- Goal: Use the product effectively, solve common problems, stay informed of changes

**Internal Teams**
- Need: PRDs (Product Requirements Documents), RFCs (Request for Comments), runbooks, design system rules, process documentation
- Goal: Make aligned decisions, onboard new team members, maintain institutional knowledge

## Main Types

**Product Documentation**
- Helps customers use products
- Covers tasks, workflows, troubleshooting, feature education
- Examples: user guides, manuals, knowledge bases, troubleshooting docs, FAQs, release notes

**Developer Documentation**
- Helps technical readers build on top of systems
- Covers APIs, SDKs, integrations, architecture
- Examples: API docs, SDK guides, integration guides, architecture docs

**Process Documentation**
- Explains how teams plan, build, review, and maintain work
- Usually internal to organizations
- Examples: project plans, PRDs, product roadmaps, RFCs, technical briefs, design systems, source code docs

## Essential Elements

A strong technical documentation page typically includes:

- **Clear purpose** — Explains what the page covers in the first few lines
- **Intended audience** — Specifies who the content is for
- **Prerequisites** — Calls out required access, tools, versions, or background knowledge
- **Step-by-step instructions** — Breaks down tasks into logical sequences
- **Examples** — Screenshots, code samples, commands, or workflows
- **Troubleshooting guidance** — Covers common errors, edge cases, solutions
- **Related links** — Helps readers continue their journey without searching
- **Ownership and maintenance** — Ensures content stays current as products change

## Why It Matters

Technical documentation:
- **Helps people succeed faster** — Reduces guesswork, improves adoption, enables self-service
- **Reduces support burden** — When people can find accurate answers, support teams spend less time on repetitive questions
- **Preserves knowledge** — Protects institutional context that could disappear through chat, tickets, absent team members
- **Improves trust and adoption** — Clear, current, searchable docs make products feel easier to use
